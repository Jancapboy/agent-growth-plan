# PRD - Agent Empire（智能体帝国）

> **文档状态**: Draft v0.1  
> **游戏类型**: 回合制模拟养成（仿《皇帝成长计划2》）  
> **主题**: AI 智能体产品从零到一的创业历程  
> **核心循环**: Sprint（双周）回合制，每回合分配有限工时处理三类核心事务

---

## 1. 游戏概述

### 1.1  elevator pitch

你是一家 AI 初创公司的 CTO。公司只有一个核心产品——一个智能体（Agent）。每两个周是一个 Sprint，你需要在有限的工时内，处理蜂拥而至的错误日志、协调混乱的数据字段、把 PM 的模糊需求翻译成精确的技术指令。同时你还要招人、选技术栈、控制预算、发布版本。三年之内，你的产品能走到哪一步？

### 1.2 核心参考

- **玩法骨架**: 《皇帝成长计划2》
  - 回合制、有限行动点、多系统并行、随机事件驱动
- **氛围参考**: 《密教模拟器》(Cultist Simulator) 的卡牌堆叠 + 资源管理
- **UI 风格**: 终端/IDE 美学 + 轻量级像素风 icon（参考 https://screenshot-to-code.vercel.app 的终端风格）

### 1.3 一句话体验

> "每个 Sprint 像开一次紧张的项目周会，但你拥有读档重来的权力。"

## 2. 核心循环：Sprint 回合

```
┌─────────────────────────────────────────────────────────────┐
│  Sprint Start                                               │
│  ├── 生成本周错误日志（奏折）                                │
│  ├── 检查字段对齐状态（外交）                                │
│  ├── 接收新需求卡片（军事）                                  │
│  └── 触发随机事件                                            │
│                                                             │
│  Player Turn: 分配 10 个工时行动点                           │
│  ├── [处理错误日志]  1-3 点                                  │
│  ├── [字段对齐]      1-2 点                                  │
│  ├── [需求公式化]    2-4 点                                  │
│  ├── [开发/编码]     1-3 点                                  │
│  ├── [招聘/面试]     1 点                                    │
│  ├── [技术调研]      1 点                                    │
│  └── [版本发布]      2 点                                    │
│                                                             │
│  Sprint End                                                 │
│  ├── 结算资源变化                                            │
│  ├── 技术债务增长/衰减                                       │
│  ├── 用户满意度更新                                          │
│  ├── 算力账单扣除                                            │
│  └── 检查结局条件                                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 行动点规则

- 每 Sprint 固定 **10 个工时行动点**
- 部分操作消耗可变（如处理 1 条日志 = 1 点，批量处理 3 条 = 2 点）
- 加班：消耗团队士气换取额外 2 点，但下一 Sprint 士气 -20%
- 周末不工作：保留 2 点未用行动点 → 下一 Sprint 士气 +10%

## 3. 四维核心资源

| 资源 | 初始值 | 范围 | 含义 | 归零/爆满后果 |
|------|--------|------|------|--------------|
| **算力 (Compute)** 💰 | 1000 | 0-∞ | 云服务预算（$/月） | 归零 = 服务宕机，游戏结束 |
| **技术债务 (TechDebt)** ⚠️ | 20 | 0-100 | 代码腐化程度 | 100 = 系统崩溃，游戏结束 |
| **用户满意度 (NPS)** 😊 | 50 | 0-100 | 产品口碑 | 0 = 用户流失，收入归零 |
| **团队士气 (Morale)** 💪 | 70 | 0-100 | 开发效率系数 | <30 = 人员离职风险 |

**资源联动公式**（每 Sprint 自动结算）：

```
算力收入 = 活跃用户数 × 付费转化率 × ARPU - 基础设施成本
技术债务增长 = 未处理错误数 × 1.5 + 需求公式化质量差的数量 × 2
用户满意度变化 = 字段对齐成功率 × 0.3 + 版本发布质量 × 0.5 - P0事故 × 15
团队士气衰减 = 连续加班 Sprint 数 × 5 + 技术债务/2
```

## 4. 三大核心系统

### 4.1 错误日志处理（奏折系统）

#### 4.1.1 机制

每 Sprint 开始时，系统生成 **3-7 条错误日志**（取决于技术债务水平）。玩家需要逐一"批阅"。

#### 4.1.2 错误日志卡片结构

```typescript
interface ErrorLog {
  id: string;
  level: 'INFO' | 'WARN' | 'ERROR' | 'CRITICAL';  // 对应奏折紧急程度
  source: string;  // 哪个模块：API网关/RAG检索/LLM调用/数据库/...
  message: string; // 错误摘要，带点技术梗
  age: number;     // 已存在多少 Sprint，越久越严重
  
  // 处理选项及后果
  options: {
    quickFix: { cost: 1, techDebtDelta: +2, npsDelta: 0 };      // 临时补丁
    properFix: { cost: 2, techDebtDelta: -5, npsDelta: +2 };    // 正经修复
    ignore: { cost: 0, techDebtDelta: +age*2, npsDelta: -level*3 }; // 搁置
    delegate: { cost: 0, requires: 'senior_dev', techDebtDelta: -3 }; // 委派
  };
}
```

#### 4.1.3 错误日志示例

| 级别 | 来源 | 摘要 | 处理建议 |
|------|------|------|----------|
| WARN | RAG检索 | `chunk_overlap` 设为 0.9，召回率 12%，用户问"怎么退款"得到「公司成立于2018年」 | properFix 或 ignore（会恶化） |
| ERROR | LLM调用 | `context window exceeded`，用户长对话第 47 轮被截断 | quickFix（截断策略）或 properFix（换模型） |
| CRITICAL | API网关 | 循环依赖：Agent A 调用 Tool B → Tool B 回调 Agent A，死锁 8 分钟 | 必须处理，ignore 直接 +15 TechDebt |
| INFO | 数据库 | 字段 `user_id` 在 users 表是 INT，在 logs 表是 VARCHAR，JOIN 时隐式转换 | 字段对齐系统的入口 |

#### 4.1.4 特殊：错误日志的连锁反应

- 如果同一模块连续 3 Sprint 出现 ERROR+ 未处理 → 触发 **P0 事故事件**
- P0 事故：NPS -20，算力消耗 ×2（回滚+紧急扩容），强制消耗 3 工时处理

### 4.2 字段对齐（外交系统）

#### 4.2.1 机制

智能体需要对接 **N 个外部数据源**（模拟真实世界的数据孤岛）。每个数据源有自己的字段命名规范和格式。

#### 4.2.2 数据源卡片

```typescript
interface DataSource {
  id: string;
  name: string;        // "用户中心" / "订单系统" / "CRM" / "埋点平台"
  fields: Field[];     // 该源提供的字段
  reliability: 0-100;  // 数据质量
  integrationStatus: 'unconnected' | 'connected' | 'aligned';
}

interface Field {
  name: string;        // 该源内部的字段名
  type: 'STRING' | 'INT' | 'FLOAT' | 'JSON' | 'TIMESTAMP';
  semantic: string;    // 实际含义："用户唯一标识" / "订单创建时间"
  sample: string;      // 示例值
}
```

#### 4.2.3 对齐玩法

当两个数据源存在**语义相同但命名/格式不同**的字段时，触发对齐事件：

```
🚨 字段冲突 detected!

数据源 A (用户中心):  user_id  INT      示例: 10042
数据源 B (订单系统):  uid      VARCHAR  示例: "U-10042"
语义: 用户唯一标识

请选择对齐策略：
[1] 手动映射 (1 工时) — 精确但不可扩展
[2] 引入中间层 (2 工时 + 100 算力) — 长期方案，新增数据源自动适配
[3] 强制转换 (0 工时) — 运行时 cast，可能丢数据
[4] 推迟处理 (0 工时) — 暂时用 fallback 值
```

#### 4.2.4 对齐状态影响

| 状态 | 效果 |
|------|------|
| 未对齐 | 该字段查询返回 null/fallback，用户满意度 -2/Sprint |
| 手动映射 | 正常，但新增数据源时需重新处理 |
| 中间层 | 正常，新增数据源自动适配，+5 NPS |
| 强制转换 | 90% 正常，10% 触发数据错误日志 |

#### 4.2.5 高级：Schema 演化

每 5-10 Sprint，一个已对齐的数据源会**变更 Schema**（模拟真实世界 API 升级），需要重新对齐。玩家可以提前投资 "Schema 监控"（技术调研）来提前预警。

### 4.3 需求公式化（军事系统）

#### 4.3.1 机制

PM/业务方每 2-3 Sprint 提出一个**模糊需求**。玩家需要将其"公式化"为精确的技术指令，然后投入开发。

#### 4.3.2 需求卡片结构

```typescript
interface Requirement {
  id: string;
  title: string;           // 业务表述："让 Agent 更智能"
  description: string;     // 详细描述（充满模糊词）
  urgency: 'low' | 'medium' | 'high' | 'critical';  // 老板关注度
  
  // 公式化过程
  formulaization: {
    steps: FormulaStep[];  // 需要完成的拆解步骤
    quality: 0-100;        // 当前公式化质量
  };
}

interface FormulaStep {
  name: string;            // "明确智能的定义" / "确定评估指标" / "设计技术方案"
  cost: number;            // 工时消耗
  skillRequired: string;   // 需要什么人才
  qualityContribution: number; // 完成后对最终质量的贡献
}
```

#### 4.3.3 需求公式化示例

**原始需求**：
> "咱们 Agent 现在回答太机械了，要更有人情味。最好像真人客服那样，能感知用户情绪，然后调整语气。这个应该不难吧？我看 OpenAI 的 demo 挺简单的。"

**公式化步骤**（玩家需依次执行）：

1. **明确"人情味"的量化指标** (1 工时)
   - 选择：情绪识别准确率 / 回答多样性 / 用户情感评分
   - 选错 → 最终质量 -20

2. **技术方案设计** (2 工时，需要架构师)
   - 选项 A: 微调 LLM (高成本，高质量)
   - 选项 B: Prompt Engineering (低成本，中等质量)
   - 选项 C: 外挂情绪识别模型 (中等成本，可能增加延迟)

3. **评估指标定义** (1 工时)
   - 必须定义可量化的成功标准，否则上线后无法评估

4. **开发实现** (消耗剩余工时)
   - 公式化质量 × 投入工时 = 最终功能质量

#### 4.3.4 公式化质量的影响

| 质量区间 | 效果 |
|----------|------|
| 90-100 | 功能超预期，NPS +10，可能出圈 viral |
| 70-89 | 正常交付，NPS +5 |
| 50-69 | 勉强可用，NPS +1，技术债务 +5 |
| 30-49 | 用户吐槽，NPS -5，技术债务 +15 |
| 0-29 | 功能回滚，NPS -15，算力浪费 ×2 |

#### 4.3.5 特殊：紧急需求插入

随机事件：**老板突然要求下周 demo 给客户看**
- 跳过公式化步骤，直接开发
- 质量锁定在 30-50
- 但如果拒绝，老板满意度下降，可能影响融资

## 5. 辅助系统

### 5.1 人才招聘（科举系统）

```typescript
interface Candidate {
  name: string;
  role: 'junior' | 'mid' | 'senior' | 'staff' | 'principal';
  skills: string[];        // "LLM微调" / "RAG优化" / "类型系统" / "DevOps"
  salary: number;          // 每 Sprint 消耗算力
  trait: string;           // 特殊特质
}
```

**特质示例**：
- "代码洁癖"：修复错误日志时 TechDebt -2 额外
- "Prompt 巫师"：需求公式化质量 +15
- "数据侦探"：字段对齐成功率 +20%
- "996 拥护者"：士气不随加班下降，但离职时带走 2 个同事
- "Rust 传教士"：坚持重构核心模块到 Rust，消耗 5 Sprint 但 TechDebt -30

**招聘流程**：
1. 发布 JD（消耗算力）
2. 简历筛选（1 工时）
3. 面试（1 工时，可能发现隐藏特质）
4. 发 Offer（成功则加入团队）

### 5.2 技术栈建设（后宫系统）

选择一个核心框架作为"正宫"，影响全局加成：

| 技术栈 | 加成 | 代价 |
|--------|------|------|
| LangChain | 开发速度快 +30%，文档丰富 | 抽象泄漏严重，TechDebt 增长 +10% |
| LlamaIndex | RAG 场景最强，字段对齐 +20% | 生态相对小，招人难度 +1 |
| 自研框架 | 无 TechDebt 惩罚，完全可控 | 每功能多消耗 50% 工时 |
| Dify | 低代码，需求公式化快 | 定制化受限，高级需求质量 -20 |

可以"纳妾"（引入辅助工具）：向量数据库、监控平台、CI/CD 等，每个提供特定加成。

### 5.3 版本发布（出征系统）

每 4-6 Sprint 可以进行一次版本发布：
- 小版本（Hotfix）：1 工时，修复紧急 bug
- 中版本（Feature）：2 工时 + 已完成的需求
- 大版本（Major）：3 工时 + 技术栈升级

发布成功 → NPS +10-30，可能吸引媒体报道（算力收入 +50% 持续 2 Sprint）
发布失败（回滚）→ NPS -20，算力 ×3（紧急回滚成本）

### 5.4 融资事件

当 NPS > 70 且 TechDebt < 40 时，可能触发融资事件：
- 天使轮：算力 +2000，但新增投资人董事（每 5 Sprint 强制要求一个"战略性需求"）
- A轮：算力 +10000，团队必须扩张到 10 人+，管理复杂度上升

## 6. 随机事件系统

每 Sprint 结束时，根据当前状态触发 0-2 个随机事件：

| 事件 | 触发条件 | 效果 |
|------|----------|------|
| **核心员工离职** | Morale < 40 | 损失 1 个 senior，TechDebt +10（交接成本） |
| **竞品发布** | NPS > 60 | NPS -10，但如果 TechDebt < 30 可快速跟进 |
| **Hacker News 首页** | 版本发布质量 > 90 | 算力收入 ×3 持续 3 Sprint |
| **云服务涨价** | 算力 < 500 | 基础设施成本 +50%，迫使你优化 |
| **开源社区贡献** | 自研框架 + 发布高质量版本 | 免费获得 2 个候选人简历 |
| **监管审查** | 用户量 > 10000 | 必须完成"数据合规"字段对齐，否则罚款 |
| **LLM 提供商宕机** | 使用第三方 LLM | 所有 LLM 相关功能不可用 1 Sprint |
| ** prompt 泄露** | TechDebt > 60 | 竞争对手复制你的核心 prompt，NPS -15 |
